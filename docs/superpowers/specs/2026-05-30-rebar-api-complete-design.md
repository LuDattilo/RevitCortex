# RevitCortex Complete Rebar API Design

**Date:** 2026-05-30
**Status:** Approved design, pending implementation plan
**Owner:** RevitCortex

## Goal

Add a complete, practical RevitCortex MCP surface for Revit reinforcement workflows: shape-driven rebar, free-form rebar, area reinforcement, path reinforcement, fabric reinforcement, couplers, splices, constraints, propagation, visibility, bending details, numbering, rounding, and reinforcement settings.

The implementation must preserve the project rules:

- Every plugin command returns `CortexResult<object>`.
- Every destructive or model-changing command uses `session.RequestConfirmation(...)`, with dry-run support where a preview is meaningful.
- All C# changes build for `Debug R25` and `Debug R24`; release work also validates R23, R26, and R27.
- Revit 2023/2024 net48 compatibility takes precedence over newer syntax or APIs.
- Tools use language-independent identifiers where possible: element ids, `OST_*` categories, type ids, and enum names.

## Compatibility Strategy

The baseline implementation targets APIs available in Revit 2023/2024 whenever possible. Newer APIs are isolated behind compile-time guards:

- `BarTerminationsData` and the newer `Rebar.CreateFromCurves(..., BarTerminationsData, ...)` overload are Revit 2026+ only.
- `RebarSpliceUtils` is available from Revit 2025+.
- `ReinforcementUtils` is available from Revit 2024+.
- `RebarBendingDetail` is available from Revit 2024+.

Tool schemas should remain stable across versions. If a caller requests a feature unavailable in the active target, the plugin returns `CortexResult.Fail(CortexErrorCode.InvalidInput, ...)` with a suggestion that names the minimum Revit version.

## Tool Surface

### Module 1 - Discovery and Inventory

Read-only tools:

- `list_rebar_bar_types`
- `list_rebar_hook_types`
- `list_rebar_shapes`
- `list_rebar_cover_types`
- `list_rebar_crank_types`
- `list_rebar_splice_types`
- `list_rebar_fabric_types`
- `get_rebar_host_data`
- `get_rebar_element_data`
- `get_rebar_geometry`
- `get_rebar_constraints`
- `get_reinforcement_settings`
- `get_rebar_api_capabilities`

These tools return compact summaries by default. Detailed geometry and per-bar data must be opt-in with `includeGeometry`, `includePerBarData`, or `maxBars`.

### Module 2 - Shape-Driven and Free-Form Rebar

Write tools:

- `create_rebar_from_curves`
- `create_rebar_from_shape`
- `create_free_form_rebar`
- `set_rebar_layout`
- `set_rebar_shape`
- `set_rebar_hooks`
- `set_rebar_terminations`
- `set_rebar_host`
- `set_rebar_visibility`
- `move_rebar_in_set`
- `include_exclude_rebar_bars`
- `split_rebar`

`create_rebar_from_curves` accepts bound line and arc curve specs in millimeters. For R26+ it uses `BarTerminationsData` when termination data is supplied. For R23-R25 it uses the older hook overloads and returns a clear error if the requested termination mode cannot be represented.

`create_free_form_rebar` supports unconstrained free-form curves and documented built-in server GUID paths where available. It does not accept arbitrary user-provided server code.

### Module 3 - Area, Path, and System Reinforcement

Write tools:

- `create_area_reinforcement`
- `create_path_reinforcement`
- `set_area_reinforcement_layers`
- `set_path_reinforcement_options`
- `convert_rebar_system_to_rebars`
- `remove_rebar_system`

Read-only tools:

- `get_area_reinforcement_data`
- `get_path_reinforcement_data`

Area reinforcement supports host-boundary creation and explicit boundary curves. Path reinforcement supports primary and alternating bar shapes, bar orientation, hooks, flip, and shape validation.

### Module 4 - Fabric Reinforcement

Write tools:

- `create_fabric_area`
- `create_fabric_sheet`
- `place_fabric_sheet`
- `set_fabric_sheet_bend_profile`
- `remove_fabric_reinforcement_system`

Read-only tools:

- `get_fabric_area_data`
- `get_fabric_sheet_data`
- `get_fabric_wire_data`

Fabric tools cover `FabricArea`, `FabricSheet`, `FabricSheetType`, `FabricWireType`, `FabricWireItem`, sheet mass, sheet location, wire centerlines, bend profiles, host placement, lap splice options, cover offset, and tag-capable views.

### Module 5 - Constraints, Propagation, Couplers, and Splices

Write tools:

- `manage_rebar_constraints`
- `propagate_rebar`
- `create_rebar_coupler`
- `set_rebar_coupler_visibility`
- `splice_rebar`
- `unify_rebars`
- `remove_rebar_splice`
- `transfer_rebar_annotations`

Read-only tools:

- `get_rebar_constraint_candidates`
- `get_rebar_coupler_data`
- `get_rebar_splice_data`
- `get_rebar_splice_candidates`

`manage_rebar_constraints` uses action-based operations: `list_handles`, `list_current`, `list_candidates`, `set_preferred`, `remove_preferred`, `recompute`, `set_to_surface`, and `highlight`. It does not expose raw `Reference` serialization as a primary public contract; references are resolved from supported target descriptors such as host face, cover, and other rebar targets.

Splice tools are version-gated to Revit 2025+. In R23/R24 they return a structured unsupported-version error.

### Module 6 - Settings, Numbering, Rounding, and Bending Details

Write tools:

- `set_reinforcement_settings`
- `manage_rebar_rounding`
- `manage_fabric_rounding`
- `manage_rebar_numbering`
- `create_rebar_bending_detail`
- `modify_rebar_bending_detail`

Read-only tools:

- `get_rebar_rounding`
- `get_fabric_rounding`
- `get_rebar_numbering`
- `get_rebar_bending_detail_data`

Numbering uses Revit numbering schema APIs for rebar where available: list sequences, remove gaps, shift numbers, move sequences, merge sequences, append sequences, and change number. Bending details are version-gated to Revit 2024+.

## External Server APIs

The API contains `IRebarUpdateServer`, `IRebarSpliceServer`, and proxy classes for custom extension servers. RevitCortex will not compile arbitrary server implementations from MCP input. That would require add-in lifecycle registration and would bypass the safety model.

Coverage for these APIs is handled by:

- `get_rebar_api_capabilities`: reports server-based APIs as available-but-not-runtime-scriptable.
- `create_free_form_rebar`: supports Revit's registered/built-in free-form server GUIDs where they are discoverable and safe.
- A future extension point can add first-party compiled servers, but not user-submitted C# server code.

## Input Contracts

Coordinates and distances are millimeters in MCP inputs unless a field explicitly says otherwise. Revit internal feet are never exposed as the primary user-facing unit.

Curve specs:

```json
[
  {"type": "line", "start": {"x": 0, "y": 0, "z": 0}, "end": {"x": 0, "y": 0, "z": 3000}},
  {"type": "arc", "start": {"x": 0, "y": 0, "z": 0}, "mid": {"x": 500, "y": 0, "z": 500}, "end": {"x": 1000, "y": 0, "z": 0}}
]
```

Layout specs:

```json
{
  "rule": "fixed_number",
  "number": 10,
  "arrayLengthMm": 1500,
  "barsOnNormalSide": true,
  "includeFirstBar": true,
  "includeLastBar": true
}
```

Version-gated termination specs:

```json
{
  "start": {"type": "hook", "hookTypeId": 12345, "orientation": "Right", "rotationDegrees": 0},
  "end": {"type": "none"}
}
```

## Result Contracts

Creation tools return:

- created element ids
- resolved type ids and names
- host id
- applied layout
- warnings
- `versionLimitations` when a request was degraded for an older Revit target

Read tools return:

- compact summaries first
- optional detailed arrays capped by explicit `max*` parameters
- all distances in millimeters
- enum names as strings

Failures use existing `CortexErrorCode` values:

- `InvalidInput` for bad ids, unsupported enum values, invalid geometry, unavailable version, or invalid host
- `ElementNotFound` for missing rebar, type, host, shape, hook, coupler, or system element
- `PermissionDenied` for read-only mode or unsafe server-code requests
- `TransactionFailed` for failed Revit transactions
- `Cancelled` when the user cancels the confirmation dialog
- `Unknown` for unexpected Revit API exceptions after contextual details are captured

## Safety and Read-Only Mode

Read-only tools must use allowed prefixes: `get_`, `list_`, or `analyze_`.

Write tools must not use read-only prefixes. They must:

- validate all ids before opening a transaction
- support `dryRun: true` for bulk or destructive changes where preview is meaningful
- request confirmation before model writes
- return counts and summaries rather than huge per-element payloads by default

Destructive operations include removing systems, converting systems to individual rebars, split/unify/splice operations, host reassignment, constraint mutation, numbering changes, and reinforcement settings changes.

## Dynamic Capabilities

Most rebar tools can be registered as always-on because they return useful errors if a project has no reinforcement. Dynamic capabilities are useful for version and model-specific hints:

- `hasRebarBarTypes`
- `hasRebarShapes`
- `hasRebarHosts`
- `hasAreaReinforcement`
- `hasPathReinforcement`
- `hasFabricReinforcement`
- `supportsRebarSpliceUtils`
- `supportsRebarBendingDetails`
- `supportsBarTerminationsData`

The MCP-visible tool list should not explode or disappear unpredictably. Prefer stable tool availability plus `get_rebar_api_capabilities` for exact availability.

## Implementation Order

Implementation should be delivered in separate, buildable steps:

1. Shared helpers plus discovery tools.
2. Shape-driven creation and layout.
3. Area/path reinforcement.
4. Fabric reinforcement.
5. Constraints, propagation, couplers, and splices.
6. Settings, numbering, rounding, and bending details.
7. Documentation, schema regeneration, and full verification.

Each step should compile and test before the next step begins.

## Open Risks

- Some operations need real Revit hosts and cannot be fully unit-tested outside Revit.
- Constraint APIs rely on `Reference` objects, which are hard to serialize safely through MCP. Public contracts must use supported target descriptors instead of arbitrary raw references.
- Rebar free-form server APIs are add-in extension infrastructure, not ordinary per-call commands. RevitCortex will expose safe first-party capabilities only.
- R23/R24 compatibility requires older overloads and careful avoidance of APIs introduced later.
- Some API calls may return null rather than throwing when geometry or shape matching fails; tools must check for null and return structured failures.

- R25 and R24 plugin builds pass.
